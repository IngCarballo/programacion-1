📝 Trabajo Práctico N°10: Despliegue de la API en la Nube
🎯 Objetivo
Desplegar la API completa (Django + DRF) en un servicio en la nube, probarla con endpoints reales y documentar el proceso de publicación. Se deben adjuntar capturas de la API funcionando (swagger/redoc, endpoints clave, health check, etc.).

📦 Entregables
URL pública de la API (ej.: https://mi-api-production.up.railway.app).


Repositorio con la configuración de despliegue (Procfile, Dockerfile o Service config).


Capturas de:


Swagger/Redoc (drf-spectacular) abierto en producción.


Respuesta exitosa de /api/health/ (o similar).


Endpoint protegido accedido con credenciales reales (200/401 según corresponda).


Panel del proveedor mostrando el servicio “Running”.


Documento paso a paso “Cómo se subió” (ver plantilla al final).


(Opcional) Colección Postman/Insomnia exportada probando endpoints productivos.



✅ Criterios de aprobación
La API responde públicamente (200 OK) a endpoints abiertos.


Endpoints protegidos devuelven 401 sin credenciales y 200 con sesión/token válido.


Documentación Swagger/Redoc accesible en producción.


Código de despliegue reproducible (quien corrige puede repetirlo).


Capturas incluidas y bien identificadas.



🧰 Requisitos previos
Proyecto Django/DRF funcionando localmente.


drf-spectacular configurado (o similar) para /api/schema/swagger-ui/.


Base de datos: PostgreSQL.


.env local con variables (no subirlo al repo).



☁️ Opciones de despliegue (elegí una)
Railway: simple, rapido de configurar y compatible con PostgreSQL. (Recomendado para este TP)


Fly.io: con Docker; flexible y rápido.


Render: muy comodo para PostgreSQL + static build.
